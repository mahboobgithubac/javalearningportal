Java Multithreading Interview Questions and Answers

Prepare for interviews with essential multithreading concepts, examples, and concurrency tools

🟢 Beginner-Level

1. What is a thread in Java?

A thread is the smallest unit of execution. In Java, threads can run concurrently to perform multitasking.

2. How do you create a thread in Java?

// Extending Thread
class MyThread extends Thread {
    public void run() {
        System.out.println("Thread running...");
    }
}

// Implementing Runnable
class MyRunnable implements Runnable {
    public void run() {
        System.out.println("Runnable running...");
    }
}

new MyThread().start();
new Thread(new MyRunnable()).start();

3. What is the difference between start() and run()?

  • start() creates a new thread and invokes run().
  • run() executes in the current thread like a normal method.

4. Thread Lifecycle in Java

New → Runnable → Running → Blocked/Waiting → Terminated

5. What are daemon threads?

Background threads that exit when all user threads finish.

Thread t = new Thread(() -> System.out.println("Daemon"));
t.setDaemon(true);
t.start();

10. What are ThreadLocal variables?

ThreadLocal is a special class in Java used to create variables that are local to each thread. Each thread accessing such a variable (via its get or set methods) has its own, independently initialized copy of the variable.

✅ Use Case: When you want to avoid shared data between threads and ensure thread-safety without using synchronization.

Basic Syntax:


ThreadLocal<Type> myThreadLocal = new ThreadLocal<>();
myThreadLocal.set(value);   // Set value for current thread
Type value = myThreadLocal.get();  // Get value for current thread

Example: ThreadLocal with Simple Counter


public class ThreadLocalExample {
    static ThreadLocal<Integer> threadLocalCounter = ThreadLocal.withInitial(() -> 0);

    public static void main(String[] args) {
        Runnable task = () -> {
            threadLocalCounter.set(threadLocalCounter.get() + 1);
            System.out.println(Thread.currentThread().getName() + " -> " + threadLocalCounter.get());
        };

        Thread t1 = new Thread(task, "Thread-1");
        Thread t2 = new Thread(task, "Thread-2");

        t1.start();
        t2.start();
    }
}

Advantages of ThreadLocal

  • Provides thread confinement of variables.
  • Reduces need for synchronization and locks.
  • Improves performance in multithreaded environments.
  • Useful in frameworks (like Spring, Hibernate) for storing user sessions or transaction contexts.

Common Methods

  • set(T value) – sets the value for the current thread.
  • get() – gets the value for the current thread.
  • remove() – removes the value for the current thread.
  • withInitial(Supplier) – Java 8 method to initialize default value.

Important Notes

  • Each thread holds an independent copy of the variable.
  • Can lead to memory leaks if not removed properly in thread pools (call remove()).
  • Not suitable for sharing data between threads — use `synchronized` or concurrent collections instead.
ThreadLocal local = ThreadLocal.withInitial(() -> 1);

🔵 Advanced-Level

11. What is ExecutorService?

Used to manage thread pools and run async tasks.

ExecutorService is an advanced framework in Java to manage and control threads efficiently. It is part of the java.util.concurrent package and provides a high-level API for launching and managing asynchronous tasks.

✅ Why use ExecutorService?
It decouples task submission from the details of how each task will be run, including thread use, scheduling, and more.

Common Methods in ExecutorService

  • submit(Runnable task) – submits a task for execution and returns a Future.
  • execute(Runnable task) – executes the task but doesn’t return a result.
  • shutdown() – initiates an orderly shutdown where previously submitted tasks are executed but no new tasks will be accepted.
  • shutdownNow() – attempts to stop all actively executing tasks.
  • awaitTermination(long timeout, TimeUnit unit) – waits for all tasks to complete or time to expire.

Basic Example using ExecutorService


import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

public class ExecutorExample {
    public static void main(String[] args) {
        ExecutorService executor = Executors.newFixedThreadPool(2);

        Runnable task1 = () -> System.out.println("Task 1 running in " + Thread.currentThread().getName());
        Runnable task2 = () -> System.out.println("Task 2 running in " + Thread.currentThread().getName());

        executor.execute(task1);
        executor.execute(task2);

        executor.shutdown();
    }
}

Types of Thread Pools

  • Executors.newFixedThreadPool(int n) – a pool with fixed number of threads.
  • Executors.newCachedThreadPool() – creates new threads as needed and reuses previously constructed ones.
  • Executors.newSingleThreadExecutor() – single worker thread for sequential task execution.
  • Executors.newScheduledThreadPool(int n) – supports scheduled and periodic task execution.

Example using submit() and Future


import java.util.concurrent.*;

public class FutureExample {
    public static void main(String[] args) throws Exception {
        ExecutorService executor = Executors.newSingleThreadExecutor();

        Callable<String> task = () -> {
            Thread.sleep(1000);
            return "Result from Callable";
        };

        Future<String> future = executor.submit(task);

        System.out.println("Waiting for result...");
        String result = future.get(); // blocks until result is available
        System.out.println("Received: " + result);

        executor.shutdown();
    }
}

Best Practices

  • Always shutdown the executor to free resources.
  • Prefer using submit() with Callable for result-returning tasks.
  • Use Future.get() carefully as it can block indefinitely unless timeout is provided.
  • Consider using ScheduledExecutorService for periodic tasks.
⚠️ Tip: Avoid using raw threads for complex applications. Always prefer ExecutorService for better performance, scalability, and maintainability. 
ExecutorService ex = Executors.newFixedThreadPool(2);
ex.submit(() -> System.out.println("Run"));
ex.shutdown();

11. What is the Runnable interface?

Runnable is a functional interface used to define tasks with no return value.
ExecutorService is a higher-level replacement for managing threads, which can execute Runnable tasks asynchronously.

Key Methods of ExecutorService with Runnable

  • void execute(Runnable command) – Executes the given command at some time in the future.
  • Future<?> submit(Runnable task) – Submits a Runnable task for execution and returns a Future representing that task.

Example 1: Using execute() method

import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

public class RunnableExample {
    public static void main(String[] args) {
        ExecutorService executor = Executors.newFixedThreadPool(2);

        Runnable task = () -> {
            System.out.println("Task is running by: " + Thread.currentThread().getName());
        };

        executor.execute(task);
        executor.shutdown();
    }
}

Example 2: Using submit() method

import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;
import java.util.concurrent.Future;

public class RunnableWithSubmit {
    public static void main(String[] args) throws Exception {
        ExecutorService executor = Executors.newSingleThreadExecutor();

        Runnable task = () -> {
            System.out.println("Running task using submit");
        };

        Future<?> future = executor.submit(task);

        // You can check if the task is completed
        System.out.println("Task done? " + future.isDone());

        executor.shutdown();
    }
}

When to use Runnable with ExecutorService?

  • When your task doesn't need to return a result.
  • For better thread management compared to raw threads.
  • To scale your application using thread pools.

17. Semaphore, CountDownLatch, CyclicBarrier

  • Semaphore: controls access with permits
  • CountDownLatch: waits for tasks to complete
  • CyclicBarrier: waits for threads to reach barrier

🔐 Semaphore

Semaphore controls access to a resource by multiple threads using permits.

Semaphore semaphore = new Semaphore(2); — Allows 2 threads at a time.

import java.util.concurrent.Semaphore;

public class SemaphoreExample {
    public static void main(String[] args) {
        Semaphore semaphore = new Semaphore(2); // Only 2 threads allowed at a time

        Runnable task = () -> {
            try {
                semaphore.acquire();
                System.out.println(Thread.currentThread().getName() + " acquired permit.");
                Thread.sleep(1000);
                System.out.println(Thread.currentThread().getName() + " releasing permit.");
                semaphore.release();
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        };

        for (int i = 0; i < 5; i++) {
            new Thread(task).start();
        }
    }
}

⏳ CountDownLatch

CountDownLatch allows one or more threads to wait until a set of tasks complete.

CountDownLatch latch = new CountDownLatch(3); — Main thread waits until 3 tasks call countDown().

import java.util.concurrent.CountDownLatch;

public class CountDownLatchExample {
    public static void main(String[] args) throws InterruptedException {
        CountDownLatch latch = new CountDownLatch(3);

        Runnable worker = () -> {
            System.out.println(Thread.currentThread().getName() + " working...");
            latch.countDown();
        };

        for (int i = 0; i < 3; i++) {
            new Thread(worker).start();
        }

        latch.await(); // Main thread waits here
        System.out.println("All tasks completed. Main thread continues.");
    }
}

🔁 CyclicBarrier

CyclicBarrier lets a set of threads wait for each other to reach a common barrier point.

CyclicBarrier barrier = new CyclicBarrier(3); — All 3 threads must call await() before proceeding.

import java.util.concurrent.BrokenBarrierException;
import java.util.concurrent.CyclicBarrier;

public class CyclicBarrierExample {
    public static void main(String[] args) {
        CyclicBarrier barrier = new CyclicBarrier(3, () -> {
            System.out.println("All parties arrived at barrier. Let's proceed!");
        });

        Runnable task = () -> {
            try {
                System.out.println(Thread.currentThread().getName() + " waiting at barrier");
                barrier.await(); // Wait for others
                System.out.println(Thread.currentThread().getName() + " passed the barrier");
            } catch (InterruptedException | BrokenBarrierException e) {
                e.printStackTrace();
            }
        };

        for (int i = 0; i < 3; i++) {
            new Thread(task).start();
        }
    }
}

🔍 Summary

  • Semaphore: Controls the number of threads accessing a resource.
  • CountDownLatch: Waits for operations to complete before proceeding.
  • CyclicBarrier: Synchronizes threads at a common execution point.

27. Difference between Thread Pool and ForkJoinPool?

  • ThreadPool: Ideal for independent tasks
  • ForkJoinPool: Ideal for recursive, divide-and-conquer tasks

🔸 ThreadPoolExecutor

java.util.concurrent.ThreadPoolExecutor

✅ Best Suited For:

  • Tasks that are independent and don’t need to fork into subtasks.
  • Traditional server applications, web request handling, etc.

🔧 Key Characteristics:

FeatureDescription
Task TypeRunnable (or Callable) tasks
ExecutionEach task is submitted and executed by a thread
Thread ReuseThreads are reused to avoid overhead
QueueingUses a BlockingQueue to hold tasks
WorkloadSuitable for uniform, independent tasks
CustomizationHighly configurable (core size, queue, etc.)

🔁 Example:

ExecutorService executor = Executors.newFixedThreadPool(4);
executor.submit(() -> {
    // Some independent task
});

🔸 ForkJoinPool

java.util.concurrent.ForkJoinPool

✅ Best Suited For:

  • Recursive, divide-and-conquer tasks
  • CPU-intensive operations with subtasks

🔧 Key Characteristics:

FeatureDescription
Task TypeForkJoinTask (e.g. RecursiveTask)
ExecutionUses work stealing to balance load
Fork/JoinTasks split (fork) and merge (join) results
Threading ModelWorker threads with work-stealing queues

🔍 Summary Table

Feature ThreadPoolExecutor ForkJoinPool
Main Task Type Runnable / Callable RecursiveTask / RecursiveAction
Use Case General-purpose thread reuse Divide-and-conquer parallelism
Work Queue BlockingQueue Deque with Work Stealing
Execution Model Submit and forget Fork, compute, and join

✅ When to Use Which?

  • Use ThreadPoolExecutor when:
    • Tasks are independent and unrelated.
    • You need manual control over threads.
    • Used in web servers, database calls, etc.
  • Use ForkJoinPool when:
    • Tasks can be split into smaller subtasks.
    • Best for parallel processing like sorting, matrix multiplication.

✅ Atomic Classes in Java

Atomic classes are part of the java.util.concurrent.atomic package and provide lock-free, thread-safe operations on single variables or data structures using low-level atomic CPU instructions (like CAS - Compare-And-Swap).

🔸 Why Use Atomic Classes?

  • Avoid use of synchronized blocks
  • Ensure non-blocking thread safety
  • High performance in multithreaded applications

🔧 Commonly Used Atomic Classes

Class Description Example Use
AtomicInteger For atomic operations on int values Counters, indexes
AtomicLong Atomic operations on long values Large counters
AtomicBoolean Atomic operations on boolean Flags, switches
AtomicReference<V> Atomic updates to object references Shared config/data
AtomicIntegerArray Atomic array of int values Concurrent numeric array
AtomicLongArray Atomic array of long values Concurrent numeric array
AtomicReferenceArray<E> Atomic array of object references Shared cache or object pool

📘 Code Examples

✅ AtomicInteger Example

import java.util.concurrent.atomic.AtomicInteger;

public class AtomicCounter {
    private AtomicInteger counter = new AtomicInteger(0);

    public void increment() {
        counter.incrementAndGet(); // atomically increases by 1
    }

    public int getValue() {
        return counter.get();
    }
}

✅ AtomicBoolean Example

import java.util.concurrent.atomic.AtomicBoolean;

public class OneTimeAction {
    private AtomicBoolean hasRun = new AtomicBoolean(false);

    public void runOnce() {
        if (hasRun.compareAndSet(false, true)) {
            System.out.println("Running one-time setup...");
        }
    }
}

✅ AtomicReference Example

import java.util.concurrent.atomic.AtomicReference;

class SharedData {
    String value;
}

public class ReferenceExample {
    private AtomicReference<SharedData> ref = new AtomicReference<>();

    public void updateData(SharedData newData) {
        ref.set(newData);
    }

    public SharedData getData() {
        return ref.get();
    }
}

⚙️ Key Atomic Operations

Method Purpose
get()Get current value
set(x)Set new value
lazySet(x)Eventually sets the value
compareAndSet(expected, new)Set if current equals expected
getAndIncrement()Increment and return old value
incrementAndGet()Increment and return new value

accumulateAndGet(x, func)Custom accumulation logic

🧠 Internals

  • Internally uses Unsafe operations
  • Performs CAS (Compare-And-Swap)
  • No locks → better scalability and reduced contention

📌 Use Cases

  • Counters (AtomicInteger, AtomicLong)
  • One-time initialization (AtomicBoolean)
  • Shared reference updates (AtomicReference)
  • Lock-free data structures and state machines

🔐 Object-level Lock

Object-level lock is used to synchronize a non-static method. It locks the current instance of the object, so only one thread can access the synchronized method per instance.

🔐 Class-level Lock

Class-level lock is used to synchronize a static method or block. It locks the Class object associated with the class, affecting all instances of the class.

🧪 Example

class LockExample {

    public synchronized void objectLevelLock() {
        System.out.println(Thread.currentThread().getName() + " acquired object-level lock");
        try { Thread.sleep(1000); } catch (InterruptedException e) {}
    }

    public static synchronized void classLevelLock() {
        System.out.println(Thread.currentThread().getName() + " acquired class-level lock");
        try { Thread.sleep(1000); } catch (InterruptedException e) {}
    }
}

public class TestLock {
    public static void main(String[] args) {
        LockExample obj1 = new LockExample();
        LockExample obj2 = new LockExample();

        // Object-level lock (per instance)
        new Thread(() -> obj1.objectLevelLock(), "Thread-1").start();
        new Thread(() -> obj2.objectLevelLock(), "Thread-2").start();

        // Class-level lock (shared across all instances)
        new Thread(() -> LockExample.classLevelLock(), "Thread-3").start();
        new Thread(() -> LockExample.classLevelLock(), "Thread-4").start();
    }
}

📊 Comparison Table

Feature Object-level Lock Class-level Lock
Applied on Instance methods Static methods or synchronized(Class.class)
Lock target Specific object instance Class object (shared by all instances)
Effect scope Only one thread per object instance Only one thread for the entire class
Usage public synchronized void method() {} public static synchronized void method() {}

📝 Conclusion

  • Use object-level lock when you need synchronization per object instance.
  • Use class-level lock when shared data among all instances must be synchronized.

🧵 Difference Between wait() and sleep() in Java

📌 Overview

Both wait() and sleep() cause the current thread to pause execution, but they are used in different contexts and have different effects.

📊 Comparison Table

Feature wait() sleep()
Defined in java.lang.Object java.lang.Thread
Needs synchronization? Yes, must be called from synchronized block or method No synchronization required
Releases monitor? Yes No
Wakes up by notify() / notifyAll() After time expires or interrupted
Interruptible? Yes Yes
Use case Thread coordination / inter-thread communication Pause execution for specified time

🧪 Example of sleep()

public class SleepExample {
    public static void main(String[] args) {
        System.out.println("Thread sleeping for 2 seconds...");
        try {
            Thread.sleep(2000);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
        System.out.println("Thread woke up");
    }
}

🧪 Example of wait() and notify()

class Shared {
    synchronized void waitMethod() throws InterruptedException {
        System.out.println("Thread waiting...");
        wait(); // releases the lock
        System.out.println("Thread resumed");
    }

    synchronized void notifyMethod() {
        notify();
    }
}

public class WaitExample {
    public static void main(String[] args) {
        Shared shared = new Shared();

        Thread t1 = new Thread(() -> {
            try {
                shared.waitMethod();
            } catch (InterruptedException e) {}
        });

        Thread t2 = new Thread(() -> {
            try {
                Thread.sleep(2000);
                shared.notifyMethod();
            } catch (InterruptedException e) {}
        });

        t1.start();
        t2.start();
    }
}

🎯 Interview Points

  • sleep() is used for pausing execution; wait() is used for thread coordination.
  • wait() must be called inside a synchronized context.
  • wait() releases the monitor; sleep() does not.
  • Both can throw InterruptedException.

✅ Summary

If you're coordinating threads to wait for a condition, use wait() and notify(). If you're simply delaying execution, use sleep().
